# Temporal Grounding and Verification Framework

**A framework for preventing temporal hallucinations in Large Language Models through temporal knowledge graph retrieval and post-generation verification.**

**Zero paid APIs required — runs entirely on your local machine.**

---

## Research Problem

Large Language Models frequently generate temporally incorrect information: wrong dates, wrong entity-time associations, reversed event orderings, and impossible temporal relationships. Existing Retrieval-Augmented Generation (RAG) systems retrieve relevant documents but do not verify whether the generated output is temporally consistent with known facts. Current hallucination detectors classify claims as "supported" or "unsupported" without distinguishing temporal errors from other types of factual mistakes.

This framework fills that gap by retrieving temporal evidence from a structured knowledge graph, grounding LLM reasoning in that evidence, and then independently verifying that every claim in the generated output is temporally correct before presenting it to the user.

## System Architecture

```
User Question
       |
       v
[1] Query Parser
    Extract entity names and temporal expressions (spaCy NER + regex)
       |
       v
[2] Temporal Retriever
    Query in-memory knowledge graph for relevant timestamped facts
       |
       v
[3] LLM Reasoner
    Generate answer grounded in retrieved evidence (Qwen3 30B via LM Studio)
       |
       v
[4] Claim Extractor
    Decompose answer into individual (subject, predicate, object, timestamp) claims
       |
       v
[5] Temporal Verifier + Hallucination Detector
    4-check verification: entity validity, relation validity,
    timestamp accuracy, temporal ordering correctness
       |
       v
[6] Explanation Generator
    Evidence-based justifications for each verification decision
       |
       v
Final Verified Answer + Confidence Score + Explanations
```

## Novel Contributions

1. **Temporal Hallucination Detection Framework** -- Four-category classification (Supported, Partially Supported, Unsupported, Temporally Impossible) instead of binary supported/unsupported
2. **Post-Generation Temporal Verification Layer** -- Independent verification after LLM generation, catching errors that context injection misses
3. **Temporal Hallucination Benchmark** -- A new dataset of correct temporal facts paired with controlled corruptions (timestamp, entity, relation, ordering) -- no such benchmark exists in published research
4. **Hybrid Retrieval + Verification Architecture** -- Retrieval constrains generation; verification catches what slipped through
5. **Temporal Consistency Scoring** -- Quantitative scoring on a continuous scale from fully consistent to completely inconsistent

## Quick Start

### Prerequisites

- **Python 3.11+**
- **LM Studio** ([download](https://lmstudio.ai)) with any model loaded (we use Qwen3 30B)

### Installation

```bash
# 1. Clone the repository
git clone https://github.com/47combinator/Temporal-Grounding-for-Preventing-Temporal-Hallucinations-in-Large-Language-Models.git
cd Temporal-Grounding-for-Preventing-Temporal-Hallucinations-in-Large-Language-Models

# 2. Install Python dependencies
pip install -r requirements.txt

# 3. Download spaCy English model
python -m spacy download en_core_web_sm

# 4. Start LM Studio server (from command line)
lms server start
```

### Run the Full Pipeline

One command does everything — downloads data, builds the knowledge graph, generates the benchmark, runs verification, and tests LLM integration:

```bash
python run_all.py
```

This will:
1. **Download** ICEWS14 dataset (90,730 temporal facts)
2. **Build** in-memory knowledge graph (no database installation needed)
3. **Generate** 20,050 hallucination benchmark examples (4 corruption types)
4. **Run verification** and compute accuracy/precision/recall/F1
5. **Test LLM** connection to LM Studio
6. **Run sample queries** with LLM-grounded temporal Q&A

### Output Files

| File | Description |
|------|-------------|
| `data/processed/knowledge_graph.json` | Full temporal knowledge graph |
| `data/benchmarks/temporal_hallucination_benchmark.json` | 20,050 benchmark examples |
| `results/verification_results.json` | Accuracy, precision, recall, F1 scores |

## How the Verification Engine Works

Every claim generated by the LLM is independently verified through four checks:

| Check | What It Does | Example |
|-------|-------------|---------|
| **Entity Verification** | Does this entity exist in the knowledge graph? | "Does 'United States' exist?" |
| **Relation Verification** | Has this relation ever been recorded between these entities? | "Did USA ever 'Make statement' to Russia?" |
| **Timestamp Verification** | Does the claimed timestamp match the recorded timestamp? | "Did it happen on 2014-03-15 or a different date?" |
| **Ordering Verification** | Is the claimed temporal ordering correct? | "Did Event A really happen before Event B?" |

Results are classified into five categories:

| Status | Meaning |
|--------|---------|
| **Supported** | All checks pass |
| **Partially Supported** | Entities and relation correct, timestamp close but not exact |
| **Unsupported** | Entity exists but relation or timestamp is wrong |
| **Temporally Impossible** | Claimed ordering contradicts recorded timestamps |
| **Cannot Verify** | Entity not found in the knowledge graph |

## LM Studio Setup

This project uses **LM Studio** as the local LLM server. Any model works, but we tested with **Qwen3 30B**.

### Starting the Server

```bash
# Option 1: Command line (recommended)
lms server start

# Option 2: LM Studio GUI
# Open LM Studio -> Settings -> Developer -> Enable Developer Mode
# Then go to Developer tab -> Local Server -> Status should show "Running"
```

The server runs at `http://localhost:1234/v1` (OpenAI-compatible API).

### Changing the Model

Edit `.env` to use a different model:

```env
OLLAMA_BASE_URL=http://localhost:1234/v1
OLLAMA_MODEL=your-model-name-here
```

## Temporal Hallucination Benchmark

The benchmark is generated from ICEWS14 using four corruption strategies:

| Strategy | What Changes | Example |
|----------|-------------|---------|
| Timestamp corruption | Replace date with random different date | (USA, MakeStatement, Russia, **2014-03-15**) -> (USA, MakeStatement, Russia, **2014-09-22**) |
| Entity corruption | Replace subject or object with random entity | (**USA**, MakeStatement, Russia, 2014-03-15) -> (**China**, MakeStatement, Russia, 2014-03-15) |
| Relation corruption | Replace relation with random different relation | (USA, **MakeStatement**, Russia, 2014-03-15) -> (USA, **Threaten**, Russia, 2014-03-15) |
| Order corruption | Swap timestamps between two related events | Event A on March 1, Event B on July 15 -> Event A on **July 15**, Event B on **March 1** |

Every corruption is checked against the full knowledge graph to ensure it does not accidentally match a real fact.

## Evaluation Metrics

| Metric | What It Measures |
|--------|-----------------|
| **Hallucination Detection Accuracy** | Percentage of benchmark examples correctly classified |
| **Temporal Consistency Score (TCS)** | Fraction of claims in an answer that are temporally correct |
| **Precision** | Among flagged hallucinations, how many are real |
| **Recall** | Among real hallucinations, how many are caught |
| **F1 Score** | Harmonic mean of precision and recall |
